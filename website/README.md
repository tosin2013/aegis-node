# Aegis-Node website

Public documentation site for Aegis-Node, built with [Docusaurus 3](https://docusaurus.io/)
and deployed to [Vercel](https://vercel.com/). The docs source lives in the
sibling [`docs/`](../docs/) tree — this directory contains only the site shell
(theme, landing page, blog, configuration).

## Local development

```bash
cd website
npm install
npm run start          # http://localhost:3000
npm run build          # production build into website/build/
npm run typecheck      # tsc, no emit
```

`npm run start` does not pre-render every page, so it's fast for iteration but
won't surface every broken link. Run `npm run build` before opening a PR.

## Where content lives

| Content                      | Source                                                   |
|------------------------------|----------------------------------------------------------|
| Operator + reference docs    | [`../docs/*.md`](../docs/)                               |
| Architectural Decision Records | [`../docs/adrs/`](../docs/adrs/)                       |
| Phase plans + research       | [`../docs/plans/`](../docs/plans/), [`../docs/research/`](../docs/research/) |
| Blog posts                   | [`blog/`](./blog)                                        |
| Landing page                 | [`src/pages/index.tsx`](./src/pages/index.tsx)           |
| Sidebar structure (Diataxis) | [`sidebars.ts`](./sidebars.ts)                           |
| Site config                  | [`docusaurus.config.ts`](./docusaurus.config.ts)         |

To add a new top-level guide, drop a markdown file in `../docs/` and add its
slug to `sidebars.ts`. To add a new ADR, drop the file in `../docs/adrs/` — the
ADRs category sidebar is autogenerated and will pick it up.

## Vercel deployment

Production lives at https://aegis-node-docs.vercel.app (Vercel project
`decision-crafters-projects/aegis-node-docs`).

### Git integration (set up once via the dashboard)

1. *Add New Project* → import `tosin2013/aegis-node` from GitHub.
2. **Root Directory:** set to `website`.
3. **Framework Preset:** Docusaurus 2 (the same preset covers Docusaurus 3).
4. **Build Command:** `npm run build` (default).
5. **Output Directory:** `build` (default).
6. **Install Command:** `npm install` (default).
7. **Node.js Version:** 20.x or later.

With **Root Directory = `website`** set, Vercel still checks out the full
repository for Git-triggered builds and runs `npm run build` inside `website/`.
The Docusaurus config reads docs content from `../docs/` and that path
resolves correctly because the full repo is present.

Once linked, every push to `main` deploys to production and every PR gets a
preview deploy automatically. `vercel.json` in this directory locks the build
settings so Vercel doesn't drift from the values above.

### CLI deploys (manual / hotfix)

The Docusaurus config reads docs from the sibling `../docs/` tree. `vercel`
CLI uploads only the current working directory, so a plain `vercel --prod`
run from `website/` will fail at build time with *"The docs folder does not
exist for version 'current'. A docs folder is expected to be found at
`../docs`"* — the parent directory simply isn't in the upload.

The fix is to build locally (where `../docs/` exists) and upload only the
prebuilt output:

```bash
cd website
vercel pull --yes --environment production       # one-time per env, refreshes .vercel/
vercel build --prod                              # runs the docusaurus build into .vercel/output
vercel deploy --prebuilt --prod                  # uploads the prebuilt output, skips remote build
```

For preview deploys, replace `--prod` with no flag (or `--target=preview`)
in both `vercel build` and `vercel deploy`.

This asymmetry only affects CLI deploys; the Git integration is unaffected
because it uploads the whole repo.

## CI

The [`website` workflow](../.github/workflows/website.yml) runs `npm ci` +
`npm run build` on PRs that touch `website/**` or `docs/**`. Vercel handles the
actual deploy — CI exists to fail PRs fast when the site can't build.

## Diataxis structure

The sidebar is organised by the [Diataxis](https://diataxis.fr/) quadrants:

- **How-to guides** — `INSTALL`, `CHAT`, `MODEL_MIRRORING`, `SUPPLY_CHAIN`
- **Reference** — `COMPATIBILITY_CHARTER`, `COMPLIANCE_MATRIX`
- **Explanation — Architecture decisions** — every ADR
- **Explanation — Plans & research** — phase plans + security research briefs

Tutorials (a learning-oriented top-to-bottom walkthrough) aren't represented
yet. The existing [`examples/`](../examples/) directory is the closest thing —
add tutorial-style markdown to `../docs/` when the content is written, then
list it under a new `Tutorials` category in `sidebars.ts`.

## Conventions

- **Markdown only.** `docusaurus.config.ts` sets `markdown.format = "md"` so
  existing GitHub-flavored content is not parsed as MDX. Don't introduce JSX
  into `.md` files — if you need components, use `.mdx` explicitly.
- **Broken links warn, don't fail.** Existing docs were written for GitHub
  with relative repo links. Build-time link cleanup is a follow-up. Treat new
  broken-link warnings in CI as defects to fix.
- **No analytics or trackers.** Aegis-Node is a security-review-oriented
  project; a docs site that phones home would be off-brand. Don't add an
  analytics plugin without an ADR.
