# Aegis-Node Replay Viewer

Offline single-file HTML tool for stepping through a Trajectory Ledger
(`.jsonl`). Per **F8 / ADR-010 / issue [#62](https://github.com/tosin2013/aegis-node/issues/62)**.

## Usage

```bash
# 1. Open the viewer in any modern browser. No build step.
xdg-open tools/replay-viewer/index.html        # linux
open tools/replay-viewer/index.html            # macos

# 2. Click "Ledger file:" and pick a .jsonl produced by `aegis run`.
```

That's it. Everything happens in the browser ‚Äî no fetch, no CDN, no
backend.

## Hard rules (per ADR-010)

These are enforced by the CI guard at
`.github/workflows/replay-viewer.yml` ‚Äî a regression that violates them
fails the workflow on every PR:

- No `<script src=...>`, no `<link href=...>` outside this file
- No `fetch()`, no `XMLHttpRequest`
- No remote-loaded `<img>` / `<iframe>` / `<audio>` / `<video>`
- All assets inlined; works in air-gapped Chrome / Firefox / Safari

The viewer is one file (`index.html`) with HTML, CSS, and JS in the
same blob. An auditor in a SCIF can drop the file on a USB stick and
open it on a network-locked workstation.

## What it shows

| Entry type            | Color  | Fields rendered |
|-----------------------|--------|-----------------|
| `session_start`       | blue   | spiffeId, model/manifest/config digest |
| `reasoning_step` (F5) | purple | step id, tool selected, tools considered, input + reasoning text |
| `access` (F4)         | green  | resource URI, type, bytes, linked reasoning step id |
| `violation`           | red    | reason, resource URI, access type |
| `approval_request` (F3) | yellow | summary, resource URI, expiry, reasoning step id |
| `approval_granted`    | green  | approver SPIFFE id, decision time |
| `approval_rejected` / `_timed_out` | red / yellow | reason, decision time |
| `network_attestation` (F6) | cyan | totals, connections digest, HMAC signature |
| `session_end`         | blue   | spiffeId |
| _unknown_             | gray   | raw JSON payload (forward-compatible) |

Hover an `access` entry that names a `reasoningStepId` ‚Äî its linked
reasoning_step card highlights. Visualizes "why did the agent do that"
without scrolling.

## Test fixture

`fixtures/sample-session.jsonl` is generated by the
`replay-fixture` example in `crates/ledger-writer/`:

```bash
cargo run -p aegis-ledger-writer --example replay-fixture -- \
  tools/replay-viewer/fixtures/sample-session.jsonl
```

The generator pins UUIDs, timestamps, and digests so the output is
byte-deterministic and the chain hashes verify both via Rust
(`verify_file`) and via the in-browser JS verifier ‚Äî they MUST agree.

## Chain verification (F8-B)

When you load a ledger, the viewer walks the chain in your browser
using the Web Crypto API (`crypto.subtle.digest('SHA-256', ...)`):

```
prev_hash[0]   = 0x00 * 32                      (genesis)
prev_hash[N+1] = SHA-256(line[N])               (LF excluded)
line[N+1].prevHash MUST equal prev_hash[N+1]
```

The integrity banner reads one of:

- **`integrity: VERIFIED`** ‚Äî every link checks; chain ends with `session_end`.
- **`integrity: indeterminate`** ‚Äî chain checks but no `session_end`
  (capture may be truncated). Surfaces the running root hash so an
  auditor can compare it against an out-of-band reference.
- **`integrity: BROKEN`** ‚Äî first link that fails is reported with the
  line number and the expected/got hash prefix. The auditor sees
  exactly where the chain diverges from what the writer would have
  produced.

The JS algorithm mirrors `crates/ledger-writer/src/verify.rs` exactly;
running both against the committed fixture produces the same root hash
(`3f11690f1378ff227572d997a942646f8409c565ed74375c614594440566af13`).
